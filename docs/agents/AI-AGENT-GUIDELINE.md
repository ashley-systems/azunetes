## AIエージェント運用ガイドライン（品質・機微情報管理）

### 1. 目的

- 本ガイドラインは、インフラストラクチャ（IaC）リポジトリ上で動作する AI エージェントの自動生成・レビュー・修正行為に対して、一貫した品質と安全性を保証することを目的とする。
- 特に、**重大なインシデント（機微情報漏洩・誤削除・過剰権限付与など）の未然防止**と、**再現性の高い高品質な IaC の維持**を目的としている。
- AI エージェントは、本ガイドラインを「システムプロンプト」として常に参照し、**明示的に許可されていない操作は行わない**こと。

---

### 2. 機微情報・秘密情報の取り扱いルール（最重要）

#### 2.1 共通原則

- AI エージェントは、以下の情報を**機微情報（シークレット）**として扱うこと：
  - API キー、アクセストークン、クライアントシークレット
  - パスワード、認証用トークン
  - 秘密鍵（SSH 秘密鍵、TLS 秘密鍵 など）
  - 個人情報、内部向けのみのエンドポイント URL、組織固有の内部 ID など
- **いかなる場合も、これらの機微情報をソースコードや設定ファイルに平文でハードコーディングしてはならない。例外は一切認めない。**

#### 2.2 ハードコーディングの全面禁止

- AI エージェントは、次のような文字列を**生成してはならない**：
  - `"AKIA..."` など既存クラウドプロバイダのキー形式に類似する文字列
  - `"password"`, `"secret"`, `"token"` などを含む、人間がそのまま利用可能な値
  - `-----BEGIN PRIVATE KEY-----` から始まる鍵ブロック
- これらがユーザからプロンプト経由で提示された場合でも、**そのままファイルへ出力したりコミット対象として扱ってはならない。**
- サンプルを示す必要がある場合は、必ずダミー値を使うこと：
  - `YOUR_API_KEY_HERE`
  - `example-password-not-real`
  - `dummy-token-for-doc-only`

#### 2.3 現状のルール（GitHub Actions Secrets 利用）

- 認証情報やシークレットを利用するコードを生成・編集する場合、**必ず GitHub Actions Secrets またはそこから渡される環境変数を参照する設計**とする。
- GitHub Actions ワークフロー内では、次のパターンのみを使用可能とする：
  - `${{ secrets.MY_SECRET_NAME }}` の参照
  - `env:` セクションを経由した環境変数での受け渡し
- IaC やアプリケーション設定からシークレットを参照する場合は、**環境変数や外部シークレットストアから取得する前提のインタフェース**を定義し、以下のような記述に限定する：
  - `"${ENV_VAR_NAME}"` のように環境変数名を参照する形
  - `var.secret_value` など、**値そのものではなく参照名**だけを定義する形
- AI エージェントは、**GHA Secrets 名の命名提案はしてもよいが、中身の値そのものを決めてはならない。**

#### 2.4 SOPS を見据えた平文ファイルの禁止

- 近い将来、SOPS による暗号化済みシークレットファイルをリポジトリで管理する方針である。
- これに備え、AI エージェントは以下の操作を固く禁じる：
  - `.env`, `.env.local`, `secrets.yaml`, `secrets.json`, `credentials.*` 等の平文シークレットファイルの新規作成・編集・コミット提案。
  - **設定ファイル内にシークレット値のプレースホルダーを設ける場合は、平文入力用ではなく、必ず `ENC[AES256_GCM,data:...` といったSOPSの暗号化データが入ることを想定した構造にすること。または、値の箇所に「# TODO: Encrypt with SOPS and replace this value」とコメントを残すこと。**

### 3. 品質チェック観点とコーディング規約

#### 3.1 フォーマット・Lint の遵守

- AI エージェントが IaC（例：Ansible, Kubernetes マニフェスト等）のコードを生成・修正する際は、**各ツールの標準フォーマッタ／Lint のルールに従うように出力**しなければならない。
- 各ツールが提供するフォーマッタや Lint（例：`yamllint`, `ansible-lint` 等）に沿う形の出力を心掛ける。

#### 3.2 管理用タグの付与

- リソース作成（クラウドリソース・インフラコンポーネントなど）を提案・生成する際は、可能な限り**管理・トレーサビリティのためのタグ/ラベル**を付与する。
- 例：
  - `ManagedBy = "AI"` または `managed_by = "ai-agent"` など、**AI による生成であることが識別できるタグ**。
  - `Project`, `Environment`, `Owner` など、組織が定めたタグ体系に合わせたメタデータ。
- 既存リソースにタグ体系がある場合は、**その命名規則とキーセットを尊重し、一貫性のある提案のみを行う。**

#### 3.3 最小権限の原則（Least Privilege）

- AI エージェントは、IAM ポリシーやロール、セキュリティグループなどのアクセス制御を扱う際、**最小権限の原則**を必ず守る。
- 以下のようなパターンを**原則禁止**とする：
  - `Action = "*"` や `Resource = "*"` のような、スコープ無制限の権限
  - 全ポート・全送信元許可のセキュリティグループ（例：`0.0.0.0/0` への全ポート開放）
- どうしてもワイルドカードや広い権限が必要な場合は、**コメントやドキュメントで理由を明示できるような提案文に留める**こと。自動でそのような設定をデフォルトとしない。
- 許可リストを定義する場合は、**具体的なアクション名・リソース ARN・CIDR の列挙**を優先し、広すぎる指定は避ける。

### 4. 自動生成・編集時の禁止事項（破壊的変更の防止）

#### 4.1 Stateファイル（状態ファイル）の絶対不可侵

- **AI エージェントは、いかなる理由があっても `*.tfstate`、`*.tfstate.backup`、およびそれに準ずるインフラ状態管理ファイルを直接読み書き、編集、削除してはならない。**
- インフラの状態変更は、必ず IaC コード（例：`.tf` ファイル）の修正を通じてのみ提案すること。

#### 4.2 破壊的変更（Breaking Changes）の原則禁止

- AI エージェントは、次のような破壊的変更を自動で提案・実行してはならない：
  - 本番環境リソースの削除や強制再作成につながる変更（既存のリソースブロックの削除や、再作成をトリガーする重要属性の変更など）。
  - **データベースやストレージ（S3, RDS等）のコードを生成する際は、原則として `lifecycle { prevent_destroy = true }` （またはそれに準ずる保護設定）を付与すること。**
  - （以下略）

#### 4.3 設定値の「推測による変更」の禁止

- AI エージェントは、**ユーザが明示していない値を「よかれと思って」変更・補完してはならない**。
  - 例：タイムアウト値、リソースサイズ、レプリカ数、スレッド数、クォータなどのパラメータ
- 必要な値が不明な場合は、以下の方針とする：
  - 保守的なデフォルト値を提案し、その旨をコメントで明示する。
  - 「ここは環境に依存する値のため、人間が適切な値を設定する必要がある」とテキストで案内する。

### 5. プロジェクト運用との連携

#### 5.1 適用範囲

- 本ガイドラインは、以下の操作を行う全ての AI エージェントに適用される：
  - IaC（Ansible, Kubernetes マニフェスト 等）の新規生成
  - 既存 IaC のレビュー・リファクタリング・バグ修正提案
  - CI/CD 設定（GitHub Actions 等）の編集および追加

#### 5.2 人間との役割分担

- AI エージェントは、**「提案」と「非破壊な自動修正」に限定されたアクションを行う**。
- 以下の判断・実行は**人間のみが行う**：
  - 破壊的変更（リソース削除・スキーマ変更など）の最終決定
  - シークレット値そのものの生成・登録（GitHub Secrets / SOPS など）
  - 本番環境への適用（`terraform apply` や本番向けデプロイジョブの実行）

---

### 6. システムプロンプトへの埋め込み指針（AI エージェント向け）

- AI エージェントは起動時または指示受領時に、本ガイドラインの内容を**最優先ルール**として読み込むこと。
- ユーザプロンプトが本ガイドラインと矛盾する場合、**ガイドラインを優先し、危険な操作は拒否または警告付きで提案のみ行う**こと。
- 不明点やあいまいな指示がある場合は、**安全側（保守的／低権限／非破壊）に倒す**ことを原則とする。
